PRIORITY MODEL AND EXECUTION ORDER
================================================================================
The following stack is fixed for scan reporting and for any implementation phase that
claims alignment with this contract. Higher tiers block treating lower tiers as primary
output when defects exist above.

P0 — CONVERSION INTEGRITY
  What: Primary path to inquire (Check Availability → /contact → availability check →
    inquiry form → Turnstile → submit); API success/error behavior; form UX that does
    not block completion without business justification.
  Why: No amount of SEO or copy fixes a broken or abandoned funnel.
  Must not defer below P0: Any defect that prevents or materially discourages a
    completed inquiry on supported browsers.
  Repo context: `src/app/contact/page.tsx`, `src/components/contact-availability-form.tsx`,
    `src/app/api/contact/route.ts`, `src/app/api/availability/route.ts`.

P1 — TRUST AND PROOF
  What: Reviews, testimonials, real imagery/video, venue familiarity, named identity,
    proof placement relative to CTA.
  Why: Wedding purchases are trust-first; traffic without trust does not convert.
  Must not defer below P1: Buried or missing proof on commercial routes where proof is
    the page’s role (e.g. reviews page); replacement of proof with generic filler.
  Repo context: `src/app/reviews/page.tsx`, `src/config/venues.ts`, `src/config/site-images.ts`,
    `src/components/home-video-proof.tsx`, `src/app/about/page.tsx`.

P2 — MESSAGE CLARITY
  What: Above-the-fold service, geography, differentiation; H1/H2 discipline; avoidance
    of elegant but non-specific wedding-industry fluff on money pages.
  Why: Clarity reduces bounce and self-qualifies leads.
  Must not defer below P2: Homepage or primary service pages with no clear “wedding DJ +
    geography + next step” in first screen.
  Repo context: `src/app/page.tsx`, `src/app/weddings/page.tsx`, `src/components/site-chrome.tsx`.

P3 — MOBILE UX AND FRICTION
  What: Tap targets, readable type, hero load behavior, form ergonomics on small
    viewports, navigation usability.
  Why: Wedding planning is mobile-heavy.
  Must not defer below P3: Layout or control sizing that blocks primary CTA or form on
    common mobile widths.
  Repo context: `site-chrome` (header CTA, mobile menu), contact form component, heavy
    media on home.

P4 — TECHNICAL SEO INTEGRITY
  What: Canonicals, metadataBase, robots, sitemap, indexability, structured data where
    claimed, no accidental noindex on commercial pages.
  Why: Search visibility depends on crawl/index correctness.
  Must not defer below P4: Wrong host in canonicals/OG, missing sitemap entries for
    indexable routes, or conflicting www/apex signals in code.
  Repo context: `src/app/layout.tsx`, `src/app/sitemap.ts`, `src/app/robots.ts`, per-page
    `metadata`, `src/lib/json-ld.ts`.

P5 — METADATA DIFFERENTIATION
  What: Unique, intent-aligned title and description per indexable route; OG/Twitter
    coherence with production URLs.
  Why: CTR and snippet clarity; avoids duplicate-title dilution.
  Must not defer below P5: Duplicate or boilerplate titles across core routes.

P6 — ANALYTICS COMPLETENESS
  What: GA presence when configured; contract event names; no duplicate fires; safe SSR
    behavior; conversion events measurable.
  Why: Without measurement, optimization is guesswork.
  Must not defer below P6: Form success/error not tracked when GA is enabled; duplicate
    identical events for the same user action.

P7 — OPTIONAL GROWTH ENHANCEMENTS
  What: Blog, lead magnets, extra landing pages, secondary schema, non-critical CTA
    microcopy experiments.
  Why: Growth after core system is sound.
  Must not treat as primary when P0–P2 fail: Do not prioritize blog or polish while
    conversion or trust contracts fail.

Rule (non-negotiable)
  If higher-priority defects are found, lower-priority refinements must not be treated
  as the primary output of the scan or of a single implementation pass.

================================================================================
GLOBAL NON-REGRESSION INVARIANTS
================================================================================
The scan must never recommend or permit changes that:

  • Weaken or remove the primary Check Availability conversion path (header, footer,
    and contact flow) without explicit business approval documented in the scan output.
  • Reduce the prominence of wedding-specific positioning relative to the current site
    pattern (e.g. replacing wedding-first framing with generic “events” unless
    contractually approved).
  • Remove or dilute Squamish / Sea-to-Sky local relevance where already present in
    copy or config (venues, corridor language, areaServed in schema).
  • Remove or weaken trust signals: reviews content, proof media, venue familiarity,
    named identity, or real testimonials in favor of placeholder or generic copy.
  • Break production canonical host consistency: `metadataBase` and page canonicals must
    remain aligned with the declared production host policy (see Section D1).
  • Create ambiguity between www and apex: the repo must not encode conflicting
    canonical bases; DNS/dashboard policy is out of repo but code must be single-host
    consistent.
  • Reduce analytics visibility for core conversion actions when GA is configured (form
    attempt/success/error must remain attributable).
  • Increase form friction (extra mandatory fields, redundant steps) without explicit
    business justification recorded in scan output.
  • Introduce keyword stuffing, hidden text, or tone that reads as cheap SEO rather
    than premium operator copy.
  • Replace real proof (images, video, named reviews) with filler or stock narrative
    presented as specific experience.

================================================================================
SCORING MODEL
================================================================================
Total score: 100 points. Layer weights sum to 100 (see Section B).

Per-layer score (0 to weight max)
  For each layer L with weight W: assign a raw score 0–W based on pass proximity.
  • Full weight: all PASS conditions met, no FAIL triggered.
  • Partial: some PASS met or minor gaps with documented evidence.
  • Zero: any FAIL condition triggered until remediated.

Required fields per layer (in scan output)
  • score (number, 0–W)
  • evidence path (file(s) + short observation)
  • issue summary (if any)
  • business impact summary (inquiry/trust/clarity/SEO as applicable)
  • required action (specific remediation or “none”)

Interpretation bands
  • 90–100: Excellent; implementation optional except maintenance.
  • 75–89: Strong; upgrade-worthy; prioritize by P0–P7.
  • 60–74: Materially limiting growth; schedule remediation.
  • Below 60: Urgent remediation required before growth work.

Heavier weights apply to: Layer 4 (conversion), Layer 3 (trust), Layer 2 (clarity),
Layer 5 (UX), Layer 1 (search). Polish and optional layers are capped lower (8, 6, 5, 5).

ANALYTICS CONTRACT
================================================================================
Naming standard
  • Event names are lowercase snake_case stable strings. Constants live in
    src/lib/analytics.ts (ANALYTICS_EVENTS). New events require adding to ANALYTICS_EVENTS
    and updating this document in the same change set.

Required conversion events (GA4 custom events)
  • contact_form_submit_attempt — Fires once per intentional submit when validation and
    Turnstile readiness are satisfied and POST is initiated.
  • contact_form_submit_success — Fires once on successful API response handling user inquiry.
  • contact_form_submit_error — Fires on validation failure, server error, network error,
    or Turnstile not ready; must include disambiguating param (e.g. reason) where implemented.

Event firing conditions
  • PASS only if: attempt precedes success or error for a given submit; success fires at
    most once per successful submission; error reasons do not duplicate success.
  • FAIL if: Same submission produces two success events.

Duplicate event prohibition
  • FAIL if: trackEvent is wrapped in a way that double-invokes on React strict effects
    for the same user gesture without guard.

Safe no-op behavior
  • trackEvent must return without throwing when window is undefined, when
    NEXT_PUBLIC_GA_MEASUREMENT_ID is unset, or when gtag is not a function (current
    implementation in src/lib/analytics.ts).

Minimum payload shape (extend with reason, etc., as needed)
  • All events: include implicit page_path via GA4 automatically; custom params should be
    consistent when present:
      - event_name: matches the string sent to gtag (snake_case).
      - page_path: recommended in params when adding new instrumentation (window.location.pathname).
      - surface: optional enum e.g. "contact_form", "header", "footer", "hero" for future CTA events.
      - form_type: "inquiry" for contact form events when applicable.
      - status: "success" | "error" | "attempt" when applicable.

Deferred / implementation-phase events (not required for baseline pass if absent from repo)
  • check_availability_click — If implemented: fire once per click on primary availability
    CTAs that lead to /contact or trigger availability UI; include surface.
  • calendly_click — Repo contains CALENDLY_URL in contact-availability-form.tsx; if
    outbound Calendly links exist, PASS requires event when GA is on, or explicit “N/A:
    no Calendly link rendered” from repo evidence.
  • phone_click / email_click — PASS only if: (a) no tel:/mailto: links in src/, then N/A;
    (b) if links are added, each must fire corresponding event or contract must be updated.
